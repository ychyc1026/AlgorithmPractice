# 员工管理系统开发总结（C++）

## 一、项目概述

开发目标：基于 C++ 实现小型公司员工管理系统，使用多态实现职工大类与子类（外包、职员、经理、老板），通过数组存储职工信息，完成增删改查、文件持久化功能。

核心技术：C++ 多态、虚函数、文件流（fstream）、数组数据管理。

运行环境：Windows + VS Code（UTF-8 源码）+ MinGW + 旧 CMD 终端。

## 二、核心功能与设计思路

多态设计
抽象职工基类，定义纯虚函数（展示信息、计算薪资等）；
子类（外包、职员、经理、老板(该代码中只有前两种)）继承并重写虚函数，实现不同类型员工的差异化逻辑。

数据存储
内存：数组存储所有职工对象（容量100）；
持久化：文本文件保存职工信息，程序启动加载、退出 / 操作后保存。
标准流程
加载文件 → 内存操作（增删改查）→ 写回文件，每次函数操作后都写进文件一次，避免特殊卡顿情况。

## 三、开发中遇到的核心问题

### 1. 文件操作逻辑混乱（核心逻辑问题）

问题表现
多次重复调用文件读取函数，导致数据重复、计数错乱；
员工计数变量 i 频繁清零，数据丢失；
增删逻辑与文件读写耦合混乱，功能异常。

问题根源
文件读写与内存操作未分离；
每个功能函数都重复读取文件，破坏内存数据。

解决方案
文件只加载一次：程序启动时调用 loadEmp() 读取全部数据到内存；
内存优先操作：增删改查仅修改内存数组；

### 2.本文件无法直接删除 / 修改中间行

问题表现：无法直接删除文本文件中某一行员工数据。
解决方案
读取全部文件内容到内存数组；
在内存中完成修改 / 删除；
清空原文件，将修改后的内存数据重新写入文件。

### 3.控制台输出排版对不齐

问题表现：员工信息打印时，中文、数字错位，界面不整齐。
解决方案：使用格式化输出（setw、左对齐）统一字段宽度，规范排版格式。

### 4. Windows 中文乱码（终极环境问题）

问题现象
cout 控制台输出中文完全正常；
文件流 fstream 读写中文乱码，文件打开后文字畸形。
底层原理

源码VS Code 默认 UTF-8代码编写，标准现代编码
Windows 旧控制台默认 GBK(936)cout 绑定系统终端 API，自动适配编码，输出正常
文件流 fstream无终端适配，裸字节读写GBK 输入字节 + UTF-8，源码环境冲突导致文件中文乱码
完整链路
控制台输入中文 → 内存中为 GBK 原始字节 → cout 靠系统 API 兜底显示正常 → fstream 直接写入磁盘，无编码适配 → 文件保存畸形字节 → 打开乱码。

## 四、Windows 编码乱码解决方案总结

1. 常规无效方案
chcp 65001：临时生效，CMD UTF-8 支持残缺，不稳定；
setlocale：仅作用于标准库，无法控制终端编码；
UTF-8 转 GBK 代码：理论正确，受 MinGW + VS Code 终端环境限制，实际失效；
系统全局 UTF-8：风险高，导致旧软件乱码。
2. 两种稳定有效方案
源码保存为 GBK
优点：简单直接，零配置，立刻解决乱码；
缺点：不符合现代标准，跨平台（Linux/macOS）会再次乱码。

使用 Windows Terminal（推荐最优解）
微软官方新一代终端，原生默认 UTF-8；
源码 UTF-8 + 终端 UTF-8，编码完全统一；
彻底根治乱码，无需转码、无需修改系统、跨平台友好。

## 五、项目最终规范流程（最佳实践）

程序启动 → 一次性加载文件数据到内存数组；
菜单操作（增删改查）→ 仅修改内存数据；
操作完成 / 退出程序 → 数据写入文件，持久化保存；
编码统一：使用 Windows Terminal，全程 UTF-8，无乱码。

## 六、核心总结

文件操作：文本文件不能直接修改行，必须全读入内存→修改→清空重写；文件只读写一次，避免逻辑混乱。
乱码本质：VS Code UTF-8 源码 + Windows 旧 CMD GBK 终端冲突；cout 靠系统 API 正常，fstream 裸写暴露编码问题。
最优解：安装 Windows Terminal，统一 UTF-8，彻底解决中文乱码，符合现代开发规范。
